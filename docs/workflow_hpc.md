# HPC Workflow Runbook

Run the full training/evaluation/statistics workflow at scale on a SLURM
cluster, preserving the same CLI, output contracts, and reproducibility rules as
the local workflow. Snakemake submits one SLURM job per rule instance via the
SLURM executor plugin; each job runs inside the Apptainer container built from
the Nix flake.

## 1. Entry conditions

- [ ] local smoke gate passed (`just run_smoke`)
- [ ] `just dry_run` resolves the production DAG
- [ ] container image present (`DRL_env.sif`, built with `just build_sif`)
- [ ] scratch output dirs set up (§3)

## 2. Environment parity

- Same repo, same `python -m src.<name>` entrypoints as local.
- Environment pinned by `flake.lock`; container built from the flake
  (`just build_sif` → `DRL_env.sif`). The profile runs it with
  `apptainer-args: "--nv --bind /scratch"`.
- Reproducibility metadata: the git commit is captured on the login node at
  Snakemake parse time and injected into each job as `GIT_COMMIT`, so run
  metadata records the commit even though git isn't available inside the
  container. Confirm it lands non-null after a run.

## 3. Scratch storage setup (one time per clone)

Outputs (checkpoints, results, logs) are bulky; keep the repo in `$HOME`
(git-backed) and redirect the heavy output trees to Ceph `/scratch` by
symlinking them. The profile already binds `/scratch` into the container so it
can write through the links.

```bash
cd <repo>
mkdir -p /scratch/$USER/hera/{trained_model,result,logs}
ln -s /scratch/$USER/hera/trained_model trained_model
ln -s /scratch/$USER/hera/result        result
ln -s /scratch/$USER/hera/logs          logs
```

Use **absolute** symlink targets so they resolve identically inside and outside
the container. `data/splits/` is regenerated by `make_split` and stays in the
repo; it's tiny.

## 4. Submit

```bash
just run_full_slurm                      # physical_job, full pipeline
just run_full_slurm TRACE=deeplearn_job  # GPU trace
just run_full_with_base_slurm            # pipeline + baseline comparison
```

or directly:

```bash
snakemake --configfile config.yaml --profile profiles/slurm
```

Production scale (`config.yaml`): 5 seeds × 6 algorithms = 30 training jobs +
30 eval jobs, 3M steps each (`save_interval=100000 × total_saving=30`),
`n_envs=20`. `train_agent` requests a GPU (`gres=gpu:1`) and `n_envs+1` cores;
a 3M run completes in ~5–6 h.

## 5. Monitor

- `squeue -u $USER` — job status.
- Per-rule logs: `logs/snakemake/<trace>/*.log` (on scratch via the symlink).
- `just slurm_report` — SLURM efficiency report after a run.

## 6. Archive results off scratch

`/scratch` is large but may be purged; pull the analysis outputs (and the
winning algorithm's final models) into safe `$HOME` storage. The target is
idempotent (rsync), so run it **after eval** to snapshot the aggregation/stats
inputs early, and **again at the end** for stats + models.

```bash
just archive_results                     # -> $HOME/drl_archive/<trace>/
just archive_results TRACE=deeplearn_job ARCHIVE=/some/other/path
```

It copies `result/<trace>/` (eval_runs, aggregate, stats, best, baseline,
plots) plus the manifests, and — once `select_best` has run — the final
checkpoints of the winning algorithm (matched by `treatment_id` in
`logs/run_log.csv`).

## 7. Output validation

- [ ] all 30 training + 30 eval jobs completed (`squeue` empty, markers present)
- [ ] `result/<trace>/{aggregate,stats,best}` populated
- [ ] `result/<trace>/holdout/holdout_summary.csv` present (winner re-evaluated on holdout)
- [ ] baseline comparison present (if run)
- [ ] figures generated under `plots/`
- [ ] `git_commit` non-null in run/aggregate/stats metadata
- [ ] results archived to `$HOME`
